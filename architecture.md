This architecture plan defines how an AI agent can be introduced into OpenEMR without bypassing the security, clinical, and compliance controls that already protect patient data. The agent should live as a server-side OpenEMR module or service layer, not as a browser-only widget and not as a separate system with direct database access. Its entry points should be limited to approved user workflows, such as chart summarization, selected-document summarization, administrative message drafting, and clinician-reviewed note assistance. The agent should return draft output only. It should not independently write to the medical record, place orders, send patient messages, change medications, update billing codes, or perform clinical actions without explicit human review and approval.

The agent should access patient data through OpenEMR’s existing service layer, REST API, or FHIR API rather than querying tables directly. Each request must include the current site, authenticated user, role, patient, encounter, requested task, and purpose of use. Data retrieval should be bounded by patient context, encounter context, selected document IDs, resource type, date range, and permitted scopes. The agent should request the minimum necessary data for the task. Broad chart reads, system-wide searches, and bulk exports should be blocked unless an administrator has approved a separate backend workflow.

Authorization should be enforced before any data is retrieved and again before any generated response is saved or sent. The agent must inherit the active OpenEMR user’s permissions, ACLs, OAuth scopes, and patient context. It should never run under a shared administrator account. For API-based access, the preferred model is OAuth2/OIDC with SMART on FHIR style scopes, using patient or user context wherever possible. System-level scopes should be reserved for controlled administrative jobs and should not be available to ordinary chat or summarization features.

Responses must be verified before use. The agent should attach source references to clinical summaries, list which records were used, show retrieval timestamps, and warn when data is missing or ambiguous. Clinical, billing, prescribing, and patient-facing outputs must require human review. The system should treat AI output as assistive draft content, not truth. Unsafe, unsupported, or low-confidence responses should be blocked or clearly flagged.

Failures should be handled safely. If authorization fails, data is missing, the model times out, the vendor is unavailable, or the response cannot be verified, the agent should fail closed and show a clear non-clinical error. It should log the event without storing raw PHI unless explicitly protected by policy. Retries should be limited, rate-controlled, and auditable. Administrators should be able to disable the agent globally, per site, per role, or per workflow during incidents.

The planned stack should use OpenEMR’s PHP backend, existing APIs, a server-side AI broker, structured prompts, JSON schemas where possible, audit logging, and a HIPAA-appropriate LLM provider only after vendor and BAA review. The main tradeoff is speed versus safety. This design gives up unrestricted agent autonomy in exchange for least privilege, traceability, clinical review, and a safer path to production. Future expansion should happen only after monitoring proves the initial workflows are safe.
